## The Cross-Validation Harness (READ THIS)

**File**: `crates/pdfplumber/tests/cross_validation.rs`

The entire test suite compares Rust output against golden JSON files generated by
Python pdfplumber. Key macros:

```rust
cross_validate!(test_name, "fixture.pdf", CHAR_THRESHOLD, WORD_THRESHOLD)
cross_validate_ignored!(test_name, "fixture.pdf", "reason string")
```

Key constants:
- `CHAR_THRESHOLD = 0.95`  (95%)
- `WORD_THRESHOLD = 0.90`  (90%)
- `EXTERNAL_CHAR_THRESHOLD = 0.80`
- `EXTERNAL_WORD_THRESHOLD = 0.80`
- `COORD_TOLERANCE = 1.0`  (points)

**Matching is bidirectional CID-tolerant** (as of PR #231):
- `char_matches` accepts if either side has `(cid:N)` and positions match
- `word_matches` same

**Run the harness**:
```bash
cargo test --test cross_validation -- --nocapture
cargo test --test cross_validation -- --nocapture --include-ignored  # for ignored tests
```

**IMPORTANT**: Always `cd` into your worktree before running tests.
Wrong directory = stale binary = false results.

## Open Issues & Current State

### Issue #223 — Rotated Table Extraction
**Worktree**: `pdfplumber-rs-fix-223`
**File**: `crates/pdfplumber-core/src/table.rs`

**What we know**:
- `nics-background-checks-2015-11-rotated.pdf` (90° rotated) finds 13 cols, needs 17
- Non-rotated version finds 17 rows × 25 cols at 100%
- Chars/words/lines/rects all 100% on rotated — only table extraction fails
- Rotated page has 700H + 508V edges (correct, same as non-rotated swapped)
- **Root cause partially diagnosed**: horizontal edges at top=32.9 span `x=[42.7..588.2]`
  but edges at top=111+ span only `x=[129.2..541.7]`. The 4 outer columns (narrow ones
  at x=42.7..129.2 and x=541.7..588.2) are spanned only by the outer border line, not
  the inner body lines. This is correct PDF structure but the table algorithm needs to
  extend partial column lines to the full table extent.
- Python pdfplumber handles this via `join_tolerance` — short line segments that nearly
  meet a longer line get extended/joined.

**Acceptance criteria**: rotated nics table rate ≥ 90% (currently 5.6%)

**Key function to investigate**: `snap_edges` and `join_edges` in `table.rs`

### Issue #221 — issue-848.pdf Word Collapse + Tables
**Worktree**: `pdfplumber-rs-fix-221`
**Files**: `crates/pdfplumber-core/src/words.rs`, `crates/pdfplumber-core/src/table.rs`

**What we know**:
- Chars: 100% on all pages
- Words: 100% on even pages, ~0.6-1.1% on odd pages
- Tables: 0-6.7% on all pages (golden has 2 tables per page)
- **Root cause of word collapse**: odd pages have `dir=Rtl` chars with DECREASING x0.
  Python pdfplumber groups these as INDIVIDUAL char "words" (each char = one word at
  its own bbox). Our Rust groups them into multi-char words (correctly by visual logic,
  but wrong relative to golden).
  
  Example golden words on page 1: `'T' x0=534`, `'h' x0=528`, `'e' x0=523` — each
  letter separate. Our output: `"ehT"` (reversed) or similar grouping.
  
  The fix: for physically-Rtl pages where chars have decreasing x, DO NOT group chars
  at all OR output each char as its own word (matching Python behavior).
  Actually — Python groups some pairs ('vi', 'wi') and separates others. The rule
  is Python's standard x_tolerance=3.0 applied to the decreasing-x sequence.
  Our word extractor already has Rtl handling; the issue is our `should_split_horizontal`
  gap calculation may be wrong for Rtl.

- **Root cause of table failure**: rects are 100% matched (16/page) but table
  reconstruction fails. The 16 rects don't intersect properly to form table cells.
  Check if the rects are the actual table borders or decorative borders by comparing
  the golden table bboxes vs the rect positions.

**Acceptance criteria**: words ≥ 90%, tables ≥ 80%; convert from `cross_validate_ignored!`

## Key Design Rules for This Codebase

1. **No unsafe code**. This is a pure-safe Rust library.
2. **No new dependencies** without discussing in PR description. The crate is intentionally
   lean (lopdf, encoding_rs, serde, unicode-bidi).
3. **All coordinates are top-left origin** (y increases downward). PDF uses bottom-left.
   The y-flip happens in `char_extraction.rs`: `top = page_height - max_y`.
4. **COORD_TOLERANCE = 1.0** in cross-validation. Char positions within 1pt are considered
   matching. Word positions same.
5. **`cargo fmt` before every commit.** CI will reject unformatted code.
6. **`cargo test` must pass** before pushing. All 100 non-ignored tests must stay green.

## Known Traps (Don't Repeat These Mistakes)

- **Running tests from wrong directory**: `cargo test` from `pdfplumber-rs/` runs against
  the main worktree build, not your worktree. Always `cd` to your worktree first.
- **Debug tests in PRs**: Remove all `fn debug_*` test functions before committing.
  They bloat PRs and may fail in CI.
- **`cross_validate_ignored!` → `cross_validate!` conversion**: Only convert when you've
  actually fixed the issue AND the test passes at CHAR_THRESHOLD/WORD_THRESHOLD.
- **`page.chars()` returns `&[Char]`**: It's a slice ref. `for c in chars.iter()` not
  `for c in &chars` (double-ref compile error).
- **`TableOptions` doesn't exist**: use `TableSettings::default()`.
- **`EdgeOrientation` doesn't exist**: use `LineOrientation`.
- **`Pdf::from_bytes` doesn't exist**: use `Pdf::open(bytes.into(), None)`.
